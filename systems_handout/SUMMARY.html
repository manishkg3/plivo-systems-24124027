System Architecture & Design Summary

Generated by AI Assistant — corrected against actual sender.c / receiver.c

This document outlines the architectural decisions, system-level design, and trade-offs implemented to route a strict 20ms-cadence media stream through a hostile UDP network exhibiting packet loss, jitter, reordering, and duplication.
Known Issue — sender/receiver offset mismatch: The sender always attaches the payload from frame i-2 as the redundant copy. The receiver, however, assumes the redundant copy belongs to frame i-1, computing idx_prev = (seq - 1) % BUFFER_SIZE. This mismatch means:
  • Frame i-1 never actually gets backed up by this mechanism — its slot may get filled with frame i-2's data instead, silently corrupting content rather than leaving a clean miss.
  • Frame m's only real protection comes from a single future packet, m+2. If that packet is lost, or happens to be the packet skipped for bandwidth reasons, frame m has no backup at all.
Fix: change the receiver to idx_prev = (seq - 2) % BUFFER_SIZE to match what the sender actually transmits.

1. High-Level Architecture

The system is divided into two decoupled C applications (Sender and Receiver). The core philosophy is Proactive Redundancy over Reactive Recovery. Because a round-trip network request takes significantly longer than the 20ms frame deadline, the system ignores the optional feedback path (ports 47003/47004) entirely. Instead, it uses a custom Forward Error Correction (FEC) wire format intended to fit under a bandwidth cap.

2. Sender Design & Bandwidth Management

The sender reads standard 164-byte frames from the harness on port 47010 and multiplexes them into a larger, redundant packet sent to the relay.

  • Fixed i-2 Offset Redundancy: The sender maintains a two-frame history buffer (prev1_payload, prev2_payload). For each outgoing packet it always attaches prev2_payload — the payload from frame i-2 — as the redundant copy. It does not alternate between i-1 and i-2; i-1's payload is tracked but never transmitted.
  • Bandwidth Cap Accounting: A 324-byte packet (4-byte header + 160-byte current payload + 160-byte historical payload) is 324/164 = 1.976x the size of a plain 164-byte frame, or 324/160 = 2.025x if measured against payload bytes alone. Which of these the 2.0x cap actually refers to should be confirmed against the harness's accounting — the modulo-32 skip below is only necessary under the payload-only definition.
  • Modulo Bandwidth Cap: On every 32nd frame (seq % 32 == 0), the sender omits the redundant payload entirely and sends only 164 bytes. Averaged over 32 frames this gives (31*324 + 164)/32 ≈ 319 bytes/packet. Note this also removes the *only* backup for frame (seq - 2) at that boundary, per the offset mismatch above.

3. Receiver Design & System-Level Concurrency

The receiver is a multithreaded C application that decouples asynchronous, unpredictable network I/O from the rigid, microsecond-precision playout clock required by the harness.

Thread 1: The Network Ingest Thread

This thread sits in a blocking recvfrom loop on port 47002, absorbing the unpredictable output of the hostile relay.

  • Circular Jitter Buffer: Incoming payloads go into a shared 256-slot array, indexed via sequence_number % BUFFER_SIZE. This resolves packet reordering (payloads land in absolute slots) and packet duplication (duplicates harmlessly overwrite identical data).
  • Recovery Logic (currently mismatched): When a 324-byte packet arrives, the thread writes its trailing 160 bytes into slot (seq - 1) % BUFFER_SIZE only if that slot is empty. Because the sender actually put frame (seq - 2)'s payload there, this either misses recovering the frame it's supposed to protect, or — if slot seq-1 is genuinely empty — fills it with the wrong frame's content.

Thread 2: The Playout Timer

This thread represents the strict real-time requirement of the media player.

  • Absolute Microsecond Timing: The thread computes the exact absolute deadline for the next expected frame as T0 + DELAY_MS + (sequence * 20ms), using POSIX clock_nanosleep with CLOCK_REALTIME and TIMER_ABSTIME, waking 3ms early to avoid missing the deadline. This avoids the compounding drift of relative sleep() calls.
  • Playout Execution: On waking, the thread locks the buffer mutex, checks the current sequence slot, and if ready, wraps the payload in the 164-byte format expected by the harness and sends it to port 47020. It then clears the slot so it can be reused on the next wrap-around.

4. Trade-offs and Limitations

  • Sender/Receiver Offset Mismatch (primary risk): As detailed above, the receiver's recovery index doesn't match what the sender transmits, which can silently corrupt recovered frames and leaves each frame with only one real point of redundancy instead of an effective two-frame backstop.
  • Vulnerability Windows at the Skip Boundary: Skipping redundancy on every 32nd frame removes the one backup opportunity for whichever frame depends on that packet.
  • Burst Loss Limitations: Sustained burst drops exceeding the redundancy offset depth (2 frames) will result in permanent frame loss even once the offset mismatch is fixed.
  • No Network Feedback: The sender operates blindly and cannot adjust its redundancy strategy based on real-time congestion (e.g. Profile B's higher loss/delay).
